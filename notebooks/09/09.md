## 9. Specifying Data

As emphasized throughout this book, there is a clear separation between an AMPL *model*, which defines the structure of an optimization problem, and the *data* that provides the specific numerical values for a particular instance. Previous chapters introduced all the principal components of AMPL models: sets ([Chapters 5](../05/05.md) and [6](../06/06.md)), parameters ([Chapter 7](../07/07.md)), variables, objectives, and constraints ([Chapter 8](../08/08.md)). With those tools in hand, we can write general, reusable model descriptions. What remains is to understand how to populate those descriptions with concrete data.

In this chapter, we focus on how to supply data to AMPL models using **amplpy**, the Python interface to AMPL. Rather than writing data in the AMPL `.dat` format, amplpy allows data to be passed directly from Python using a variety of familiar data structures. This approach integrates naturally into modern data workflows: data can come from hand-coded Python literals, from computations performed earlier in the same script or notebook, or from external sources such as CSV files, spreadsheets, and databases.

Throughout the earlier chapters, we have already seen several ways to assign values to sets and parameters directly from Python. In this chapter, we revisit these ideas systematically and present the full range of practical techniques. These range from simple Python lists and dictionaries—suited to small or hand-crafted datasets—to high-performance tabular libraries such as Pandas, NumPy, and Polars, which are more appropriate when data is large, structured, or comes from an external source.

[Section 9.1](9_1_simple_data.ipynb) begins with the most fundamental operations: assigning a Python list to an AMPL set, extracting set membership from dictionary keys, and setting scalar parameter values. [Section 9.2](9_2_data_in_dicts.ipynb) shows how Python dictionaries provide a natural representation of indexed parameter data, since a dictionary's key–value pairs correspond directly to the index–value pairs of an AMPL parameter, including parameters indexed over multiple sets via tuple keys. [Section 9.3](9_3_data_composite.ipynb) extends this idea to composite data structures, where all attributes of an entity are grouped into a single tuple, yielding a compact definition from which multiple AMPL parameters can be derived with dictionary comprehensions.

When data is already available in tabular form, the methods of [Section 9.4](9_4_data_in_pandas.ipynb) are most convenient. A Pandas DataFrame can be passed to AMPL via the `set_data()` method, which simultaneously defines a set and assigns values to all parameters indexed over it in a single call. [Section 9.5](9_5_data_in_numpy.ipynb) demonstrates that NumPy arrays, which arise naturally in matrix-structured problems, fit equally well into this framework when wrapped in a DataFrame and assigned to two-dimensional AMPL parameters. For data pipelines that demand higher preprocessing performance, [Section 9.6](9_6_data_in_polars.ipynb) introduces Polars as a high-throughput alternative, using a simple conversion to Pandas as the bridge to AMPL.

Finally, [Section 9.7](9_7_data_in_external.ipynb) addresses the practical situation in which data is not hand-coded in Python but loaded from external files. We show how to read data from JSON files, CSV files, and Excel spreadsheets, and how to pass the resulting DataFrames directly into AMPL. The pipeline—external file, Pandas DataFrame, AMPL model—is a recurring pattern in real-world optimization workflows and applies uniformly regardless of the source format.

To keep the focus on data-handling methods rather than model structure, every section in this chapter uses the same underlying optimization model: the diet problem introduced in [Chapter 2](../02/02.md). This model involves a set of food items (`FOOD`), a set of nutrients (`NUTR`), and parameters for unit cost, consumption bounds, nutritional requirements, and nutrient content per serving. The use of a consistent running example makes it straightforward to compare approaches and to transfer techniques learned in one section to new settings.
